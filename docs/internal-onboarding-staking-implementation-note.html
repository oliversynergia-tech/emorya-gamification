Internal Implementation Note

Emorya Gamify: Onboarding and Staking Ladder Implementation

This note turns the onboarding and staking direction into a practical implementation view. It also reviews how much of the existing Gamify work is affected by this shift in approach.

The short answer is positive: most of the completed platform work still stands. The main changes are to Emorya-specific quest strategy, reward weighting, starter-path logic, and in-app event integration.

Working Assumption

The new approach is to treat onboarding as the main activation ladder and staking as the next monetisation-readiness ladder. This is a meaningful shift in emphasis, but not a fundamental rebuild of the product.

Estimated impact: roughly 70-80% of the implemented platform work remains directly reusable. Roughly 20-30% needs reshaping, mostly in Emorya quest content, reward design, starter-path logic, and event hooks.

What This Changes In The Product

1. The platform’s main early journey becomes more deliberate

Instead of a broad starter mix of social, app, and calorie tasks, the first visible journey becomes an activation ladder with stronger sequencing and more deliberate weighting.

2. Wallet readiness becomes more central

xPortal setup and wallet linking stop being just one quest among many. They become central milestones in the core activation path.

3. Monetisation readiness becomes a visible product stage

Staking is no longer just a trust or wallet side-task. It becomes the next meaningful progression phase after activation and retention.

Impact Review: What Stays Intact

Area Impact Why
Brand skin / partner publishing system Low Unaffected. Theme publishing and brand switching still work exactly as intended.
Custom quest builder Low The builder is already strong enough to support the new quest shapes and sequencing.
Verification lanes and review queue Low Manual review, file proof, API verification, and text submissions all remain useful.
Partner quest portability system Low Still valuable. This shift mainly affects the Emorya-native ladder, not the portability model.
Admin verification showcase and demo packs Low Still valid. These are infrastructure and admin tooling, not Emorya-specific funnel assumptions.

Impact Review: What Needs Reshaping

Area Impact What Changes
Emorya v1 launch quest set Medium The existing launch set stays useful, but the order, weighting, and emphasis need to move harder toward activation-first onboarding.
Quest board ordering Medium The current launch funnel ordering will need to be updated so the activation ladder is the dominant visible path.
Starter path logic Medium Current starter-path requirements still use generic XP/level/activity/community thresholds. These should be rebuilt around activation completion.
Emorya-specific quest metadata and XP balances Medium Several seeded quest rewards remain valid, but relative weighting needs rebalancing once onboarding becomes a premium conversion layer.
Demo data and test fixtures Medium Tests and seeded user progress should reflect the new activation milestones and not the older starter-path assumptions.

Impact Review: What Becomes More Important

Area Impact Why It Now Matters More
Real in-Emorya app event hooks High The more onboarding is central, the more valuable automatic completion from real app events becomes.
Profile completion and onboarding-state tracking High These need to exist as first-class completion states rather than vague proxies.
xPortal download/open/create journey definition High The exact wallet journey now matters because it is part of the core activation ladder.
Staking verification logic High The old single 30-day stake hold concept is no longer enough if staking becomes a full ladder with thresholds.
Commercial rules around APY boost High This needs product and business alignment before the quest ladder can be finalised properly.

Concrete Areas In The Current Build Likely To Change Next

  • seeded Emorya quest definitions in server/db/seed.sql
  • Emorya launch-set curation migrations in the later quest migrations
  • starter-path rules in lib/progression-rules.ts
  • starter-path progress rendering in server/repositories/platform-repository-db.ts
  • quest board ordering logic where the Emorya launch funnel is currently defined
  • tests tied to starter quests, progress state, and dashboard board selection

What Does Not Need Rebuilding

  • brand theme architecture
  • admin publishing of public skin
  • quest verification engine
  • custom quest builder workflow
  • proof uploads and review queue
  • API callback verification
  • partner portability metadata structure
  • milestone-share placeholders as a concept

Recommended Sequence From Here

  1. Replace the current Emorya starter path with the activation ladder.
  2. Add the missing onboarding quests: profile completion, starter setup, xPortal open/create, full activation completion.
  3. Reweight onboarding XP so the largest reward sits at full activation.
  4. Refactor the existing staking quest into a tiered staking ladder.
  5. Agree internally on APY feasibility and threshold design.
  6. Connect more real in-app events so onboarding and staking can auto-complete where possible.

Bottom Line

This switch in approach affects the product strategy meaningfully, but it does not invalidate most of the work already done. The infrastructure, admin tooling, verification model, theming, and custom quest system are all still strong and still relevant.

The main impact is that Emorya’s questing logic becomes more intentional around activation and monetisation. In practice, that means we are mostly reshaping content, sequencing, and event hooks rather than rebuilding the underlying platform.

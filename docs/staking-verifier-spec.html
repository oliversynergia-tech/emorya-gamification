Internal Spec

Staking Verifier Spec

Purpose: let Gamify auto-verify optional EMR staking quests through app or backend connection, without making staking a required gate for the core user journey.

What it must support Current stake checks, threshold checks, hold-duration checks, and later referred-staker checks.
What it must not do Staking must remain optional. It cannot block activation or the main onboarding path.

Required Source Of Truth

  • Emorya backend, MultiversX staking/indexer service, contract-query service, or equivalent internal service.
  • Must be able to answer: linked wallet address, current qualifying staked amount, qualification timestamp, and whether the stake is still active.

Verifier Input

{
  "userId": "uuid",
  "walletAddress": "erd1...",
  "questSlug": "reach-staking-threshold-a",
  "thresholdAmount": 500,
  "requiredHoldDays": 0,
  "campaignSource": "direct"
}

Verifier Output

{
  "approved": true,
  "walletAddress": "erd1...",
  "stakedAmount": 1200,
  "thresholdAmount": 1000,
  "qualifiedAt": "2026-04-02T10:15:00Z",
  "verifiedAt": "2026-04-02T10:20:00Z",
  "reason": "threshold_met"
}

Quest Decision Rules

  • stake-your-first-emr: approve if stakedAmount > 0.
  • reach-staking-threshold-a: approve if stakedAmount >= thresholdA.
  • reach-staking-threshold-b: approve if stakedAmount >= thresholdB.
  • unlock-apy-boost-status: approve if stakedAmount >= apyThreshold.
  • strengthen-the-core-30-day-hold: approve only if threshold is met, qualification date exists, 30 days have elapsed, and the stake is still active.
  • referred-staker-bonus: later, approve when a referred user also meets the required threshold.

Recommended API Shape

  • Gamify uses the existing api-check verification type.
  • Suggested endpoint: POST /staking/verify.
  • Suggested auth: shared secret or signed server-to-server token.

Fallback Behavior

  • If verification is unavailable or incomplete, return approved: false with a reason like wallet_not_found, staking_source_unavailable, threshold_not_met, or hold_period_not_met.
  • Gamify can then keep the quest incomplete or route it to manual review.

Questions For The App Team

  • Can we reliably identify the user’s linked wallet address?
  • Can we query the user’s current EMR stake amount?
  • Can we determine when the user first crossed a qualifying threshold?
  • Can we determine whether the stake remained active through the hold period?
  • Can the backend expose or call an authenticated verifier endpoint for Gamify?
Minimum viable version
Phase 1 only needs current stake amount for first stake, threshold A, threshold B, and APY threshold. Phase 2 adds hold-duration and referred-staker verification.

Short conclusion: if the app or backend can return wallet-linked stake amount, threshold status, and qualification timing, Gamify can auto-verify optional staking quests cleanly through the existing API-check flow.
